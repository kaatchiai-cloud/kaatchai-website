# Audio Rehearsal Plan — Per-Scene Audio Iteration + Pre-Video Preview Gate

Status: design draft, locked decisions, not yet implemented.
Scope: introduces a new pipeline stage between image generation and video generation where the user iterates on per-character dialogue audio per scene, previews the whole project as image-synced playback, and only then commits to expensive Kling video generation.

This plan is the natural extension of [voice-and-lipsync-plan.md](voice-and-lipsync-plan.md). Multi-voice TTS is already shipped; this plan defines **where** and **how** users review, regenerate, and lock that audio before video commitment.

Sibling to [voice-and-lipsync-plan.md](voice-and-lipsync-plan.md) and [consistency-plan.md](consistency-plan.md). Depends on the multi-voice pipeline (Phase 5) being live, which it is.

---

## 1. Goal

Three user-visible outcomes:

1. **Audio iteration on every image card.** Each scene's image card grows an inline audio player + mood selector + regen button. Users tune per-scene audio (mood, voice override, full regen) right where they review the image — no navigation break.
2. **Whole-project rehearsal preview.** A dedicated step between image-gen and video-gen plays the project as image crossfades synced to the assembled multi-voice audio track. Users see the cinematic feel before committing to expensive video generation.
3. **Audio-locks-video-duration discipline.** Once audio is finalized at rehearsal, scene durations equal audio durations. Kling clips render to that exact length. Post-video-gen audio drift > 3% blocks send-to-editor until either video is regenerated for that scene or audio is re-tuned to fit.

The architectural insight: **video is the expensive, slow, irreversible step. Audio is cheap and iterable. Lock audio first, generate video against locked audio.**

## 2. Non-goals (v1)

- Not per-character timeline lanes in the editor — Stage 1 of editor support is speaker-colored regions on the existing master dialogue track. Real per-track audio (mute/solo/volume per character) is v1.5.
- Not free-form mood text — fixed enum of 12 moods only.
- Not per-line voice cloning, voice fingerprinting from audio — those are v2.
- Not retrofitting existing projects — gate is forward-only. Projects with video already generated bypass.
- Not BGM coupling — BGM is a final-stage add-on, not a script-driven asset. It plays over the final mux at whatever total duration the audio + video produce.

Audio is generated upstream at the storyboard step (Phase 5 of the voice plan). The rehearsal flow doesn't add new generation timing — it surfaces existing audio with iteration affordances and adds a preview gate.

## 4. Pre-existing primitives reused

- **Multi-voice TTS pipeline** ([js/17b-create-references.js](js/17b-create-references.js) `castGenerateMultiVoiceAudio`): runs at storyboard step; produces per-line `AudioBuffer`s + `speakerTurns[]` metadata. **v1 ships with per-character batching with tone-aware break rule** (see §8.7 below) — supersedes voice-plan §8.3 which originally specified per-line. Voice plan to be updated to match.
- **Field name reconciliation:** voice-plan §8.4 wrote `scene.dialogueLength` in milliseconds. This plan uses `scene.audioActualDuration` in seconds as the canonical field. **Migration: voice-plan §8.4 to be updated to set `audioActualDuration = totalMs / 1000` (in seconds) instead of `dialogueLength`. The `dialogueLength` field is dropped.** Drift calculations and all downstream consumers in this plan reference `audioActualDuration` (seconds).
- **`window._createSpeakerTurns`**: live during the project — array of `{ speakerCharacterId, voiceId, startMs, endMs, segmentIndex }` matching the assembled master audio. Drives lip sync; here drives mini-player playback ranges + scene-tick positions.
  **Canonical source rule:** `window._createSpeakerTurns` is the single source of truth, populated by `castGenerateMultiVoiceAudio` and rebuilt on every regen (§6.4 step h). Per-scene `scene.speakerTurns[]` arrays mentioned in voice-plan §5.5 are DERIVED views — `scene.speakerTurns = window._createSpeakerTurns.filter(t => t.segmentIndex === scene.segmentIndex)`. They are recomputed lazily on read and never persisted independently. After regen, only `window._createSpeakerTurns` is rewritten; per-scene views automatically reflect the new state.
- **`createScenes[i].dialogue`**: per-segment dialogue object with `speakerCharacterId`, `text`, `isVoiceOver`, `actualStartMs`, `actualEndMs`. Already populated by Phase 4-5.
- **Scene image gen step** ([js/17c-create-pipeline.js](js/17c-create-pipeline.js) Step 7 / canvas SB nodes): the image-gen workspace where per-image audio mini-players land.
- **Web Audio `AudioBuffer` slicing**: standard API for cutting per-line buffers from the master combined buffer at `startMs/endMs`.
- **Wavesurfer**: already used for narration audio waveform display; same pattern reused for per-image mini-player.
- **Per-line TTS helper** (`castGenerateLineTTS`): exists; called per-regen.
- **Cost tracker** (`trackCost`): credit ledger surface for itemized regen costs.
- **`updateCreateAgentTask`**: agent-task display surface for "regenerating Maya's line 4…" progress.
- **Scene mirror updates** (`updateSceneCardImage`, `updateSceneCardStatus`): existing per-scene UI refresh entry points.
- **soundtouch-js**: WSOLA-based time-stretch library; loaded on-demand for ±3% drift handling.

### 5.1 Per-scene audio state extension

```js
// createScenes[i] additions on top of existing fields
{
  // existing: id, prompt, dialogue, framing, speakerVisible, startTime, endTime, duration, ...

  // Audio rehearsal state (this plan)
  // Three duration fields with distinct semantics:
  audioActualDuration: 3.234,         // seconds; current TTS output duration of this scene's primary audio
                                      // (dialogue or narration). Updates on every audio regen.
  manualDuration: null,               // seconds; user-set stepper value for b-roll scenes (per §6.7).
                                      // null for dialogue scenes. When narrationOverlay exists,
                                      // audioActualDuration uses narrationOverlay duration instead;
                                      // manualDuration is preserved for fallback when narration is removed.
  videoActualDuration: null,          // seconds; the actual duration of the Kling clip that was generated.
                                      // null pre-video-gen. Set when Kling returns its clip metadata.
                                      // Persists across audio regens — DO NOT overwrite on re-lock.
                                      // This is the load-bearing reference for drift calculation post-video-gen.
  duration: 3.234,                    // seconds; the "locked target" duration sent to Kling at video gen time.
                                      // Mirrored from audioActualDuration pre-lock; frozen at lock.
                                      // Distinct from videoActualDuration (Kling may produce slightly different).
                                      // Used as the time budget for Kling clip generation.
  audioStale: false,                  // true when dialogue.text or speaker changed since last TTS
  audioGeneratedAt: '<ISO>',          // when this scene's audio was last produced
  audioCharCount: 47,                 // for ElevenLabs cost recompute on regen

  // Per-line voice override (mood / voice settings for this line only).
  // dialogue describes ONE on-camera speaker turn. For scenes that ALSO have
  // narrator overlay audio (b-roll with VO over it), see narrationOverlay below.
  dialogue: {
    // existing: speakerCharacterId, speakerName, text, isVoiceOver
    voiceOverride: {                  // null = use cast default
      mood: 'angry' | 'calm' | ... ,  // from MOOD_ENUM (§5.4)
      voiceId: null,                  // null = use cast voice; set = one-off swap
      voiceProvider: null,            // null = use cast provider
      generatedAt: '<ISO>',
      stickyKey: '<hash of dialogue.text>',  // override valid only while text matches
                                              // CLEARED when override is dropped on text change (EC-IM-08)
    },
    regenCount: 0,                    // per-line cumulative regen count (drives "5+ regens" warning EC-CO-03)
    regenLockToken: null,             // multi-tab race lock — { uuid, ts } during in-flight regen, null otherwise
    muted: false,                     // user-set via editor Stage 1 context menu (EC-ED-03);
                                      // when true, line is skipped at export tick + subtitles for this line hidden
  },

  // Narrator overlay — independent of dialogue. Populated when a scene has
  // narrator voice-over PLUS some other on-camera content (dialogue OR b-roll).
  // null when scene is pure-dialogue or pure-b-roll-no-narration.
  // The image-gen card renders TWO audio sections when both dialogue + narration
  // exist: one for the speaker, one for the narrator overlay.
  narrationOverlay: null | {
    text: 'Maya finally understood what Joe meant.',  // narrator's spoken text
    voiceOverride: { ... },           // same shape as dialogue.voiceOverride
    regenCount: 0,
    regenLockToken: null,
    muted: false,
    actualStartMs: 0,                  // scene-relative ms when narrator begins.
                                       // v1 default: 0 (narration starts with scene).
                                       // Field is honored as-emitted by storyboard agent — non-zero values
                                       // supported when agent emits them (mid-scene narration onset, e.g.,
                                       // 2s of b-roll before narrator speaks). Lip sync (voice-plan §10)
                                       // uses this for sprite-paint scheduling — don't paint mouth before
                                       // narration begins.
    actualEndMs: 4500,                 // scene-relative ms when narrator stops.
                                       // narrationOverlay duration (actualEndMs - actualStartMs) may be
                                       // SHORTER than scene duration (narrator stops, ambient continues).
  },

  // Drift status — re-computed on every audio or video change
  durationStatus: 'matched' | 'stretched' | 'compressed' | 'exceeds',
  durationDriftPct: 0.021,            // (audio - video) / video; signed
  durationDriftStretched: false,      // applied time-stretch flag
}
```

### 5.2 Project-level rehearsal state

```js
window.createJobState.audioRehearsal = {
  status: 'pending' | 'reviewed' | 'locked' | 'not_applicable',
  reviewedAt: null | '<ISO>',
  lockedAt: null | '<ISO>',
  totalAudioDurationMs: 0,            // sum of audioActualDuration × 1000 across scenes; cached for display
  totalDriftPct: 0,                   // last-computed total drift; cached for banner display in §7.8
};
```

`totalVideoDurationMs` is intentionally NOT persisted at the project level — total video duration is derived on-the-fly in `canGenerateVideos()` (§7.6) from each scene's `videoActualDuration ?? duration`, avoiding a stale-cache failure mode. UI displays that need total video duration call the same on-the-fly reducer.

`hasUnresolvedScenes` was an earlier-iteration cache field; **removed** for the same reason — gate status is computed on-the-fly by `canGenerateVideos()` from per-scene `durationStatus`, not from a project-level cached flag.

**Status enum semantics:**
- `'pending'` — initial state OR audio has changed since lock; rehearsal review required again
- `'reviewed'` — user has visited rehearsal step and previewed the output; not yet locked
- `'locked'` — user has clicked "Generate videos"; audio is committed; video clips have been generated to these durations
- `'not_applicable'` — pure b-roll project (§7.9); rehearsal step is skipped

**Status revert rule (post-lock changes):**
After `status === 'locked'`, any of these actions reverts status to `'pending'`:
- User regens any scene's audio (mood, voice override, or full re-TTS)
- User edits storyboard text (triggers `audioStale`)
- User changes a character's cast-level voice
- Time-stretch fallback fails on a scene (durationStatus → `'exceeds'`)

The reverted state forces the user back through the rehearsal preview before "Generate videos" becomes available again. Existing video clips are NOT auto-regenerated — they persist; the rehearsal-step gate evaluates whether they still match the new audio (per §7.6 `canGenerateVideos()`).

**Telemetry alignment:** the boolean `audioRehearsalCompleted` in §15 is a derived view of `status === 'locked'` at the time of telemetry export, not a separate persisted flag. It can flip false again on post-lock revert.

**State machine (full diagram):**

```
                  ┌──────────────────┐
                  │   not_applicable │   pure b-roll project, no audio
                  └──────────────────┘   (terminal — never enters main flow)

  initial          ┌──────────────┐    user enters    ┌────────────┐
  ────────────────▶│   pending    │──────step─────────▶│  reviewed  │
                  └──────────────┘                    └────────────┘
                          ▲                                 │
                          │                                 │ user clicks
                          │  any audio change                │ "Generate videos"
                          │  (regen / text edit /            │ (button enabled gate
                          │   cast voice change /            │   passed per §7.6)
                          │   stretch fail) reverts          │
                          │                                  ▼
                  ┌──────────────┐                  ┌────────────┐
                  │              │◀──────audio──────│   locked   │──┐
                  │   pending    │     change       └────────────┘   │
                  │  (re-flow)   │       triggers          │         │ kicks
                  │              │                          │ video    │ off
                  │              │      ┌───────────────────┘ gen      │ Kling
                  │              │      │   completes               ──┘  for
                  │              │      ▼                                each
                  │              │  ┌─────────────────────────┐          dialogue
                  │              │  │  videoActualDuration    │          scene
                  │              │  │  set on each scene      │
                  │              │  │  (locked stays locked)  │
                  │              │  └─────────────────────────┘
                  │              │      │
                  │              │      │ audio change after
                  │              │      │ video gen
                  │              │      ▼
                  │              │◀─ pending (post-video-gen mode)
                  │              │     ▲
                  │              │     │ user re-reviews,
                  │              │     │ fixes drifts,
                  │              │     │ clicks Generate again
                  │              │     │
                  └──────────────┘     │
                          ▲             │
                          └─────────────┘
```

**Downstream artifact lifecycle on each transition:**

| From → To | Master audio buffer | Video clips (IDB/R2) | Editor timeline | Export state | Mini-players |
|---|---|---|---|---|---|
| `pending → reviewed` | reassembled from per-line buffers | unchanged | unchanged | unchanged | refresh on entry to rehearsal step |
| `reviewed → locked` | persisted to IDB as canonical | not yet generated | unchanged | unchanged | unchanged |
| `locked → (video gen kicks off)` | unchanged | created per-scene; videoActualDuration written | populated post-video-gen on send-to-editor | populated | unchanged |
| `locked → pending` (audio change) | reassembled with new buffer | **kept as-is** (regen explicit only); now mismatched until either video regen or audio re-stretch fits | timeline preserves clip references; flagged stale per scene | export disabled until re-locked | re-render affected scene's mini-player; downstream scenes shift but their mini-players unchanged |
| `pending → reviewed` (post-video) | reassembled | kept | kept | kept | refresh per drift status |
| `reviewed → locked` (post-video) | re-persisted | scenes with drift > 3% trigger Kling regen on user opt-in; their videoActualDuration is overwritten on regen completion | timeline updates clip references for regen'd scenes | re-enabled | already current |

**Editor implications when user edits audio post-video-gen:**

When user navigates from editor back to image-gen step to fix audio:
- Editor's existing timeline composition is preserved
- Audio change marks `audioRehearsal.status = 'pending'`
- A banner in editor surfaces: "Audio changed in scene N — re-run rehearsal to update"
- Send-to-editor gate from rehearsal becomes "Update editor with new audio" instead of fresh send
- User explicitly chooses what to do with stale video clips: regen for exact match, accept stretch, or revert audio

This separates the "audio is committed" question (rehearsal lock) from the "video matches audio" question (drift status), avoiding ambiguous "stuck states."

## 6. Stage 1 — Image-gen step extension (per-scene audio iteration)

### 6.1 Per-image card layout

Each scene's image card in the image-gen workspace (Step 7 / canvas SB nodes) grows three new affordances **below the existing image preview + prompt area**:

```
┌──────────────────────────────────────────────────────────┐
│  [Scene 1 image — 16:9 thumbnail]                        │
│                                                           │
│  Prompt: "Maya storms into the kitchen, shouting…"       │
│  [Edit prompt]  [Regen image]                            │
│                                                           │
│  ─── Audio ──────────────────────────────────────         │
│  🎙️  Maya · Aoede (Gemini)                                │
│  "I can't believe you said that."                        │
│  ▶ ─────●──────────────── 0:03.2                          │
│  🎭 Mood: [calm ▾]   ↻ Regen ($0.00)                      │
│                                                           │
│  Status: ✓ ready                                          │
└──────────────────────────────────────────────────────────┘
```

Three controls:

1. **Inline mini-player** — play / pause / seek through THIS scene's audio segment only. Driven by Web Audio. **Source of truth:** the per-line `AudioBuffer` cached in IDB at `audio_line_<sceneId>_<turnIdx>` (§5.5). The master combined buffer is a derived view assembled FROM these per-line buffers; the mini-player reads its own per-line buffer directly so it always reflects the latest regen for that scene without needing a master rebuild.

   Re-render trigger: when scene `i`'s audio is regen'd, only scene `i`'s mini-player re-mounts. Adjacent scenes (`i+1`, `i+2`, ...) do NOT re-mount their mini-players because their per-line buffers haven't changed — only their `startTime` in the master timeline shifted. The rehearsal step's master playback re-assembles from per-line buffers + cut-beats on each playback start; mini-players are independent of the master assembly.
2. **Mood dropdown** — fixed enum from §5.4. Pre-filled from storyboard agent's `performance.tone`, mapped via `MOOD_AGENT_MAPPING` (see §6.3a). When mapping returns no match, default is `'matter-of-fact'`. Change → marks as override; visible change to mood doesn't auto-trigger regen, just queues it; regen happens on ↻ click.
3. **Regen button** — re-runs TTS for this line with current mood + voice override. Shows cost: `$0.00` for Gemini, `$0.06` for ElevenLabs (computed from char count). Always shows cost even when zero.

### 6.2 Card state during initial generation

Storyboard step finishes → image gen runs → multi-voice TTS runs in parallel. User may land on the image-gen step before either is fully done.

```
While image is generating:
  Image preview area: skeleton spinner with "Generating…"
  Audio area:         hidden / placeholder

While audio is generating (TTS for this character's batch in flight):
  Image preview:      ready (image done)
  Audio area:         "⏳ Generating Maya's audio…" placeholder
                       (no play button until ready)

When both ready:
  Full card as in §6.1
```

The TTS batches per character — so all of Maya's lines arrive at the same time, then all of Joe's, etc. Mini-players light up character-by-character.

### 6.3 Mood selector behavior

User opens `🎭 Mood ▾` → 12-option list with icons (per §5.4). Click a mood:

1. UI updates the dropdown label immediately.
2. Card shows a `~ pending regen` badge — change is queued, audio is not yet regenerated.
3. User clicks `↻ Regen` to commit the change → fires TTS with new voice settings.

Why not auto-regen on mood change: every mood click would burn a TTS call. Explicit regen click is the cost gate.

Alternative considered: mood change auto-regens after a 1.5s debounce. Rejected — it makes cost unpredictable when user is exploring options.

### 6.3a Storyboard agent's `performance.tone` schema + mood mapping

The storyboard agent (in voice-and-lipsync-plan §7.2 the prompt extension) is asked to emit `performance: { tone: '...', gesture: '...' }` per dialogue segment. The agent's tone field is **constrained to one of the 12 MOOD_ENUM IDs** (matter-of-fact, calm, warm, serious, excited, angry, sad, whispered, playful, concerned, urgent, sarcastic).

This is enforced by updating the storyboard agent's prompt to include:

```
For dialogue segments, emit performance.tone as exactly one of:
"matter-of-fact" | "calm" | "warm" | "serious" | "excited" | "angry" |
"sad" | "whispered" | "playful" | "concerned" | "urgent" | "sarcastic"

Use "matter-of-fact" as the default when no specific emotional cue is needed.
```

When pre-filling the mood dropdown:

```js
function deriveSceneMood(scene) {
  const tone = scene.performance && scene.performance.tone;
  // Tolerant matching: agent may emit slight variations
  const lc = String(tone || '').toLowerCase().trim();
  const exact = MOOD_ENUM.find(m => m.id === lc);
  if (exact) return exact.id;
  // Loose mapping fallback for agent variations:
  const ALIASES = {
    'neutral': 'matter-of-fact', 'matter of fact': 'matter-of-fact',
    'authoritative': 'serious', 'professional': 'serious',
    'enthusiastic': 'excited', 'energetic': 'excited',
    'soft': 'whispered', 'quiet': 'whispered', 'hushed': 'whispered',
    'fun': 'playful', 'lighthearted': 'playful',
    'worried': 'concerned', 'anxious': 'concerned',
    'rushed': 'urgent', 'hurried': 'urgent',
    'ironic': 'sarcastic',
  };
  return ALIASES[lc] || 'matter-of-fact';
}
```

User overrides via the mood dropdown supersede this pre-fill. Override is stored in `dialogue.voiceOverride.mood` (per §5.1).

### 6.4 Regen flow

Click `↻ Regen ($0.06)` →

```
1. Confirm dialog (only when cost > $0.05):
   "Regen this line with current mood + voice settings?
    Cost: $0.06 (ElevenLabs, 198 chars)
    [Cancel]  [Regen]"

2. On confirm:
   a. Card transitions to "regenerating" state
      (mini-player disabled, badge shows ⏳)
   b. castGenerateLineTTS(text, voiceWithOverride) called
   c. New AudioBuffer received
   d. Old per-line buffer in master combined buffer is REPLACED
      with new buffer; subsequent scenes shift if duration changed
   e. scene.audioActualDuration updated
   f. scene.audioGeneratedAt = now
   g. scene.dialogue.voiceOverride.stickyKey = hash(dialogue.text)
   h. window._createSpeakerTurns rebuilt (boundaries shifted)
   i. Card re-renders with new mini-player
   j. trackCost('voiceRegen', cost) added to ledger
   k. autoSaveCreateState()

3. Compute the duration delta and update the regen'd scene's own endTime
   BEFORE shifting downstream scenes. Invariant: `endTime = startTime + audioActualDuration`.
   const oldAudioActualDuration = scene.audioActualDuration;   // captured before regen replaced it
   // step 3a-d above replaced scene.audioActualDuration with new TTS duration
   const deltaSec = scene.audioActualDuration - oldAudioActualDuration;
   scene.endTime = scene.startTime + scene.audioActualDuration;   // refresh regen'd scene's endTime

   Subsequent scenes' startTime + endTime shift by deltaSec.
   `scene.duration` of adjacent scenes is NOT changed (per lock — only timing slots shift):
   for (let j = i+1; j < createScenes.length; j++) {
     createScenes[j].startTime += deltaSec;
     createScenes[j].endTime += deltaSec;
     // duration unchanged; downstream per-scene drift unchanged (see step 4 clarification below)
   }

   **Cascade impact (must be surfaced to user BEFORE confirming a regen that
   changes scene duration by > 200ms when downstream scenes have already-
   generated video clips):**

   ```
   ┌──────────────────────────────────────────────────────────────────┐
   │ Regenerating scene 4's audio                                     │
   │                                                                  │
   │ New audio is 1.3s longer than current.                           │
   │                                                                  │
   │ This will:                                                       │
   │  • Shift scenes 5–30 timing by +1.3s in the master audio         │
   │  • Subtitles for scenes 5–30 will re-align to new timing         │
   │  • Scene 4's own audio-video drift will change — review in       │
   │    rehearsal. You may need to either accept time-stretch (≤3%)   │
   │    or regen scene 4's video.                                     │
   │                                                                  │
   │ Other scenes' per-scene drift is unchanged (their audio and      │
   │ video shift together).                                           │
   │                                                                  │
   │ Continue with regen?    [Cancel]    [Regen]                      │
   └──────────────────────────────────────────────────────────────────┘
   ```

   Threshold for showing the cascade warning: `Math.abs(deltaSec) > 0.2` (200ms).
   Smaller deltas don't warrant interruption.

   Status badge updates: only the regen'd scene's badge transitions
   (`✓ matched` → `⚠ stretched/compressed X%` or `⛔ exceeds X%`).
   Downstream scenes keep their pre-regen `durationStatus`.

   `audioStale` is NOT propagated to downstream scenes — their audio buffers
   haven't changed, only their position in the master timeline. Only the
   regen'd scene's `audioStale` flag is touched (set false on completion).
4. Recompute `durationStatus` and `durationDriftPct` on the regen'd scene ONLY.
   Downstream scenes' per-scene drift is **unchanged** because their
   `audioActualDuration` and `videoActualDuration` are unchanged — only their
   master-timeline `startTime`/`endTime` shifted. Per-scene drift = ratio of
   own audio to own video, so positional shift is invisible to the drift
   formula. Recomputing them is harmless but unnecessary.
5. If audioRehearsal.status was 'locked' or 'reviewed', revert to 'pending':
   if (audioRehearsal.status === 'locked' || audioRehearsal.status === 'reviewed') {
     audioRehearsal.status = 'pending';
   }
6. Increment dialogue.regenCount for telemetry / 5+ warning logic (EC-CO-03).
7. Acquire/release dialogue.regenLockToken to coordinate multi-tab race (EC-CO-04).
```

### 7.6 "Generate videos" gate

Button enabled if and only if:

```js
function canGenerateVideos() {
  // Per-scene check (your A1 lock — strict).
  // Both 'stretched' and 'compressed' are within-3% time-stretch states — they pass the gate.
  // Only 'exceeds' (>3% drift) and 'pending'/'error' states block.
  const allScenesMatched = createScenes.every(s =>
    s.durationStatus === 'matched'
    || s.durationStatus === 'stretched'
    || s.durationStatus === 'compressed'
  );

  // Total drift check (also your A1 lock).
  // Pre-video-gen: videoActualDuration is null on every scene → fall back to scene.duration
  // (which is mirrored from audioActualDuration), making total drift 0 by construction.
  // Post-video-gen: videoActualDuration is set → drift compares audio against actual Kling output.
  const totalAudio = createScenes.reduce((sum, s) => sum + (s.audioActualDuration || 0), 0);
  const totalVideo = createScenes.reduce((sum, s) => sum + (s.videoActualDuration ?? s.duration ?? 0), 0);
  const totalDriftPct = Math.abs(totalAudio - totalVideo) / Math.max(0.001, totalVideo);
  const totalOk = totalDriftPct <= 0.03;

  return allScenesMatched && totalOk;
}
```

**Three-field lifecycle:**
- **Pre-video-gen:**
  - `audioActualDuration`: live TTS-driven value; updates on every audio regen
  - `duration`: mirrored from `audioActualDuration` continuously; same value
  - `videoActualDuration`: null
  - Per-scene drift = 0 by construction; total drift = 0
- **At lock click ("Generate videos" pressed):**
  - `duration` is FROZEN to current `audioActualDuration` and sent to Kling as the clip-length budget
  - `videoActualDuration` is still null (Kling hasn't returned yet)
- **At video-gen completion (per scene as Kling returns its clip):**
  - `videoActualDuration` is populated with Kling's actual output duration (may differ slightly from `duration`)
  - From this point, `videoActualDuration` is the IMMUTABLE reference for drift calculation
- **Post-video-gen audio regen:**
  - `audioActualDuration` updates to new TTS duration
  - `duration` does NOT auto-update (it represents what was sent to Kling)
  - `videoActualDuration` does NOT change
  - Drift is computed: `(audioActualDuration - videoActualDuration) / videoActualDuration`
- **Post-video-gen re-lock (user clicks "Generate videos" again to commit a new audio set):**
  - `duration` is updated to current `audioActualDuration` for the affected scenes (these become the new Kling clip-length budget)
  - When Kling regen runs, `videoActualDuration` is overwritten with the new Kling output
  - Scenes that didn't have audio changes keep their previous `videoActualDuration` untouched

This ensures the reference point (`videoActualDuration`) is preserved across audio iterations and only updated when the user explicitly pays for video regen.

Click "Generate videos" →
1. Lock `audioRehearsal.status = 'locked'`
2. Lock `audioRehearsal.reviewedAt = now`
3. Lock per-scene `scene.duration = scene.audioActualDuration` for ALL scenes (b-roll scenes' `duration` already equals `audioActualDuration` via the continuous mirror per §7.6 lifecycle; this step formalizes the freeze for ALL scene types so Kling receives an explicit clip-length budget for every scene)
4. Persist master audio to IDB
5. Fire video-gen flow (existing pipeline)

## 8. Time-stretch + regen logic (post-video-gen)

### 8.1 When this matters

After video has been generated for a project, user can return to image-gen step and regenerate audio for a scene (different mood, voice change, etc.). The new audio's duration may differ from the existing video clip's duration.

### 8.2 Strict rule (your J lock)

| Drift magnitude | Action | Note |
|---|---|---|
| 0% – 3% | Time-stretch silently + always-visible badge + hover info | Default; honest fallback for tiny drift |
| > 3% | Hard block; only resolution = regen video for that scene | No silent stretch, no compression artifacts |

### 8.3a Drift → durationStatus truth table

Computed deterministically per scene post-video-gen. `drift = (audioActualDuration - videoActualDuration) / videoActualDuration`. Sign is preserved (positive = audio longer than video, negative = audio shorter).

```js
function computeDurationStatus(scene) {
  if (scene.videoActualDuration == null) {
    // Pre-video-gen: status panel uses audio-completion mode (§7.7)
    return scene.audioActualDuration ? 'matched' : 'pending';
  }
  const drift = (scene.audioActualDuration - scene.videoActualDuration) / scene.videoActualDuration;
  const absDrift = Math.abs(drift);
  // 0.05% threshold for "matched" — anything below the 0.1% rounding floor reads as exact
  if (absDrift <= 0.0005) return 'matched';
  if (absDrift <= 0.03) {
    return drift < 0 ? 'stretched' : 'compressed';
    // Naming convention:
    //   audio shorter than video → audio is "stretched" (made longer to fit) at playback
    //   audio longer than video  → audio is "compressed" (sped up to fit) at playback
    // The verb describes the OPERATION applied to audio, not the drift sign.
  }
  return 'exceeds';
}
```

| `audioActualDuration` vs `videoActualDuration` | drift sign | `durationStatus` | Operation applied |
|---|---|---|---|
| Equal (within 0.05%) | ~0 | `'matched'` | None |
| Audio < Video by 0.05–3.0% | negative | `'stretched'` | Audio stretched longer to fit video |
| Audio > Video by 0.05–3.0% | positive | `'compressed'` | Audio compressed shorter to fit video |
| Drift > 3.0% (either direction) | any | `'exceeds'` | None — hard block |

Implementer must use this exact mapping. Inverting "stretched"/"compressed" creates user-visible bugs.

### 8.4 Asymmetric tolerance

Stretching audio (making short audio longer) sounds better than compressing (making long audio shorter). But your J lock said: **always show indicator from 3%, regen-only above 3%** for both directions. So we treat them symmetrically at the gate level, but the hover-info text differs slightly:

```
Stretch (audio shorter than video, padding added):
  "Audio is 2.1% shorter than video. Stori inserted natural padding."

Compress (audio longer than video, sped up):
  "Audio is 2.1% longer than video. Stori compressed playback slightly."
```

Both show the same regen-for-exact-match escape hatch.

### 8.4a Degraded mode — soundtouch-js unavailable

If soundtouch-js fails to load (CDN blocked, offline, corporate firewall), the time-stretch path becomes unavailable. We can no longer hide drift via stretching, so the gate tightens automatically:

| Mode | Drift tolerance | Threshold for `⛔ exceeds` |
|---|---|---|
| Normal (soundtouch loaded) | ±3% silent stretch | drift > 3% |
| **Degraded (soundtouch failed)** | **0% — no stretch possible** | **any non-zero drift** |

When degraded mode is active, the rehearsal step shows a persistent banner:

```
⚠ Audio time-stretch library unavailable — stricter drift tolerance active
   Any audio drift will require video regeneration. Reload the page to retry
   loading the library.                                          [Reload]
```

The threshold change is significant: a project with 2.5% drift across many scenes that would pass in normal mode entirely blocks in degraded mode. Surfacing this clearly avoids the confusing "why won't my project ship?" failure where the actual cause is a missing library.

**Recovery:** Reload triggers a fresh soundtouch-js CDN attempt. If it loads, drift assessments re-run and most scenes flip from `⛔ exceeds X.X%` back to `⚠ stretched X.X%` with the gate re-enabled.

### 8.6 Subtitles after stretch

If subtitles were generated from Scribe word alignment on the original audio, their timestamps are based on the un-stretched buffer. Two valid behaviors:

- **Re-run Scribe on stretched audio.** Most accurate, costs ~$0.05.

### 8.7 Per-character batching with tone-aware break rule (v1)

This is the v1 ground truth for how `castGenerateMultiVoiceAudio` calls TTS.
Reconciles voice-plan §8.3's earlier per-line proposal with the latency
constraints discussed during plan review. **The per-character batched path
is the v1 default; voice-plan §8.3 is to be updated to match this rule.**

**Rule:** group consecutive segments by `(speakerCharacterId, mood)` —
break the batch whenever EITHER changes between adjacent segments.

```js
function planTtsCalls(segments) {
  const calls = [];   // [{ speakerId, mood, voice, segments[], joinedText }]
  let active = null;
  for (const seg of segments) {
    if (!seg.dialogue || !seg.dialogue.text) continue;
    const speakerId = seg.dialogue.speakerCharacterId || 'narrator';
    const mood = (seg.dialogue.voiceOverride && seg.dialogue.voiceOverride.mood)
      || (seg.performance && seg.performance.tone)
      || 'matter-of-fact';
    if (active && active.speakerId === speakerId && active.mood === mood) {
      active.segments.push(seg);
      active.joinedText += ' ' + seg.dialogue.text.replace(/[.!?]?$/, '.');
    } else {
      if (active) calls.push(active);
      active = {
        speakerId,
        mood,
        voice: castResolveVoiceForSpeaker(speakerId, seg.dialogue.speakerName),
        segments: [seg],
        joinedText: seg.dialogue.text.replace(/[.!?]?$/, '.'),
      };
    }
  }
  if (active) calls.push(active);
  return calls;
}
```

**Per-call output is one combined `AudioBuffer` containing all the segments
in that call.** We then split it back into per-segment buffers using Scribe
word alignment (or fall back to even-distribution timing when Scribe is
unavailable).

**Cost implications:**
- 20 dialogue lines from 3 characters with mostly-uniform mood = ~3 TTS calls
- Same 20 lines if every line has a different mood = ~20 calls (degenerates to per-line)
- Typical projects: 30–50% reduction in call count vs strict per-line, with same character-count cost on ElevenLabs (cost is char-count driven)

**Implementation note — this is a refactor, not a tweak:**

The current `castGenerateMultiVoiceAudio` at [js/17b:3115](js/17b-create-references.js#L3115) implements an **earlier per-line serial** approach (consistent with voice-plan §8.3's original v1 spec). Implementing §8.7 of this plan requires meaningful changes to that function:

| Aspect | Existing (per-line) | §8.7 target (per-character batched) |
|---|---|---|
| Iteration | One TTS call per dialogue segment | One TTS call per `(speakerId, mood)` group |
| Cut-beat between segments | Binary: 200ms different-speaker / 0ms same-speaker | Three-tier: 200ms speaker-change / 100ms same-speaker tone-change / 0ms in-batch (handled by splitter) |
| Output of each call | Single-line `AudioBuffer` (sample-accurate by construction) | Multi-line combined `AudioBuffer` requiring splitter (Tier 1/2/3 per §8.7a) |
| Boundary precision | Uniform sample-accurate | Variable per §8.7b (Tier 1: sample-accurate; Tier 3: ±200ms) |
| `cutBeatBeforeMs` field on segment audio entries | Set per-segment (current line 3146 in 17b) | Computed at master-buffer reassembly time, not per segment — splitter outputs are concatenated with explicit cut-beat insertion between groups |

**What §12 Phase 2 must include** (this is missing from §12 today and needs a sub-bullet):
- Refactor `castGenerateMultiVoiceAudio` from per-line serial loop to per-`(speakerId, mood)`-batched calls
- Replace binary `cutBeatBeforeMs` with three-tier cut-beat at master assembly time
- Implement Tier 1/2/3 splitter chain (§8.7a) and select tier based on key availability + provider
- Add regression test required by §18 acceptance (Tier-1 split + middle-line regen, ≤250ms cumulative drift)

**What stays unchanged:** `castGenerateLineTTS` ([js/17b:3069](js/17b-create-references.js#L3069)) is preserved — used directly by single-line regen (§6.4) and by single-line fallback when batching degenerates to per-line (e.g., every line has a different mood). The §8.7b boundary-precision difference between batched-and-split lines vs directly-TTS'd regen lines is a deliberate consequence of this preservation.

**Why not always per-character (no break on mood)?**
Gemini and ElevenLabs both render one consistent prosody arc per call.
Forcing "calm line then angry line" into the same call produces blended
prosody that fits neither — the angry line sounds half-calm, the calm
line sounds half-angry. Tone-aware breaks preserve the per-utterance
intent cleanly.

**Single-line regen** (user clicks ↻ on one line): always per-line, even
if the surrounding lines are batched. Replaces just that one segment's
audio buffer in the master combined buffer. Doesn't disturb the batch.

### 8.7a Splitter — primary path, fallbacks, and boundary precision

The batched-call output is a single combined `AudioBuffer`. Splitting it
back into per-segment buffers has THREE tiers, in order of accuracy:

**Tier 1 — Scribe word alignment (DEFAULT, ~95% accurate):**
This is the unambiguous default — Scribe is the source of truth for
boundary precision per voice-and-lipsync-plan §11.0. Send the combined
buffer to ElevenLabs Scribe → get per-word timestamps. For each input
line, locate its first word and last word in the timestamp stream; the
segment boundary is `[firstWord.start, lastWord.end]`. Slice the
AudioBuffer at those sample offsets. Sample-accurate; handles inter-line
silence, breath groups, and prosody drift naturally. Tier 2 and 3 are
fallback-only paths invoked when Scribe is unavailable.

**Tier 2 — ElevenLabs `*-with-timestamps` endpoint (ElevenLabs only):**
When the active provider for the batch is ElevenLabs and Scribe is
unavailable, switch the TTS endpoint from `/v1/text-to-speech/<voiceId>`
to `/v1/text-to-speech/<voiceId>/with-timestamps`. The response includes
character-level timing arrays. We map character offsets to line boundaries
(input lines were joined with `". "` separator, so each line's last
character is a `.` followed by a space → split point). Roughly 90%
accurate. Free; same call cost as standard TTS.

**Tier 3 — TTS-reported total duration + concatenated cumulative offsets
(degraded; explicit boundary spec):**
When neither Scribe nor a timestamps endpoint is available (e.g., Gemini
without Scribe), we fall back to deterministic concatenation math:

```js
function fallbackSplitByDurationMath(combinedBuffer, lines, providerReportedTotalMs) {
  // 1. Sum each line's expected duration based on word count × per-word avg.
  //    Per-word avg derived from total duration / total word count in the batch.
  const wordCounts = lines.map(l => l.text.split(/\s+/).filter(Boolean).length);
  const totalWords = wordCounts.reduce((a, b) => a + b, 0);
  const avgMsPerWord = providerReportedTotalMs / Math.max(1, totalWords);
  // 2. Allocate inter-line silence padding — a fixed 200ms gap between consecutive lines
  //    matches the cut-beat convention from voice-plan §8.4 + §8.7 single-speaker case.
  //    For same-speaker consecutive lines (pre-tone-break), padding shrinks to 100ms.
  const INTER_LINE_PAD_MS = 100;  // same-speaker, same-tone batch (typical case for §8.7)
  // 3. Compute cumulative offset per line:
  let cursorMs = 0;
  const segments = [];
  for (let i = 0; i < lines.length; i++) {
    const lineDurMs = wordCounts[i] * avgMsPerWord;
    const startMs = cursorMs;
    const endMs   = cursorMs + lineDurMs;
    segments.push({ lineIdx: i, startMs, endMs });
    cursorMs = endMs + (i < lines.length - 1 ? INTER_LINE_PAD_MS : 0);
  }
  // 4. Slice combinedBuffer at sample-accurate offsets derived from these ms values
  return segments.map(s => sliceAudioBuffer(combinedBuffer, s.startMs, s.endMs));
}
```

**Tier 3 honest accuracy:** word-rate math approximates boundaries but
doesn't account for actual prosody (a line ending in a question takes
longer than its word count suggests; a short emphatic statement can be
shorter). Boundaries may drift ±200ms per line. This is acceptable for
mini-player playback (the user hears the right line; minor lead/trail
silence is invisible) but degrades lip-sync precision (mouth-flap timing
becomes less accurate).

**When Tier 3 is active, surface a one-time hint to the user:**
"Using approximate audio boundary detection. Lip sync precision may be
slightly reduced. Configure ElevenLabs key for sample-accurate alignment."

### 8.7b Per-line buffer source — initial gen vs regen, boundary precision difference

Per-line buffers in IDB (`audio_line_<sceneId>_<turnIdx>`) come from
different code paths depending on operation:

| Operation | Per-line buffer source | Boundary precision |
|---|---|---|
| **Initial generation** (batched call → split) | Tier 1 Scribe split (best); Tier 2 / 3 fallbacks | Sample-accurate (Tier 1) → ±50ms (Tier 2) → ±200ms (Tier 3) |
| **Single-line regen** (user clicks ↻) | Direct per-line `castGenerateLineTTS` call → buffer is the entire TTS output, no splitting needed | Sample-accurate by construction (no split required) |
| **Multi-line regen** (mood change cascades to multiple consecutive lines) | New batched call → re-split via same Tier 1/2/3 chain | Same accuracy tiers as initial gen |

**Implication:** initially-generated lines and regen'd lines may have
slightly different leading/trailing silence characteristics. A regen'd
line is "tight" — it starts at sample 0 of its own buffer. An
initially-generated line from a Tier 3 split may include 50–200ms of
adjacent line's tail or its own leading silence.

**Why this matters:**
- Mini-player playback: imperceptible (a few hundred ms of silence at the
  start sounds natural; user just hears the line)
- Master combined-buffer reassembly: each per-line buffer is concat'd
  with 100ms cut-beat between same-speaker lines and 200ms between
  speaker changes. Tier-3-split lines may end up with 100ms + 100ms =
  effective 200ms gap; regen'd lines maintain exactly the cut-beat target.
  Slight rhythm difference, but uniform within a single project session.
- Lip sync: a regen'd line is more precise than its surrounding initial-
  gen lines because its own buffer has no boundary noise. Lip sync code
  should re-run face detection on the affected scene's clip after any
  regen anyway, so this doesn't compound.

**Test coverage:** the implementer should add a regression test that
runs Tier-1 split on a known sentence sequence, runs single-line regen on
the middle sentence, then confirms the resulting master combined buffer
plays back without audible cut-beat drift > 250ms total.

## 16. Risks + open questions

### Risks

1. **Mood enum coverage.** 12 moods may not cover every filmmaker request. v2 may need to expand list or allow blends. Mitigation: ship the 12 we picked; track which are used most via telemetry; expand based on data.

2. **Gemini's mood honoring inconsistency.** The `[angrily]` cue in prompt text doesn't reliably control prosody — Gemini may render it as text or partially honor. ElevenLabs voice settings are deterministic. Mitigation: surface honest expectation in UI ("ElevenLabs voices respond to mood more reliably than Gemini").

3. **Scene-boundary crossfades feel artificial in cinematic preview.** A 200ms crossfade reads as "PowerPoint slideshow," not "movie." Mitigation: configurable crossfade duration per template; "story" templates use longer (300ms) blends, "marketing" snappy cuts (50ms).

4. **Audio drift cascade.** Scene 4 regen → 0.7s longer → all subsequent scenes shift → may break send-to-editor for project that was passing. Mitigation: clear messaging ("regen of scene 4 affected scenes 5-30; review rehearsal again").

5. **Time-stretch quality on heavily-styled voices.** ElevenLabs voices with high `style: 0.8` may not stretch cleanly. WSOLA can introduce artifacts on very expressive audio. Mitigation: detect via signal analysis; fall back to "regen for exact match" suggestion automatically when stretch would introduce artifacts.

6. **Mid-project mood changes during rehearsal.** A user who hasn't seen the result yet picks a mood, hates it, picks another, hates that, etc. Each ElevenLabs cycle costs $0.06. Mitigation: surface "you've burned $0.30 regenerating Maya's line 4 — try a different mood?" warning at 5+ regens per line.

7. **Editor → image-gen navigation friction.** Per the lock, editor regen actions deep-link back to image-gen step. Editing flow break. Mitigation: only deep-link, don't auto-return; let user explicitly come back. If filmmakers find this jarring, v1.5 considers inline regen in editor.

### Open questions for v2

1. **Per-line audio editing in editor.** Trim head/tail, time-stretch one line specifically, replace one line entirely. Real DAW model. Worth 3-5 days of work if filmmakers ask.

2. **Voice cloning at cast time.** Upload an audio sample → ElevenLabs creates a custom voice → bind to character. Adds significant value for serial / brand projects but needs ToS + cost management.

3. **Mood blending.** Allow `mood: 'angry+sad'` (50/50) for nuanced delivery. Implementation: voice-settings interpolation. Probably overkill for v2; adds complexity without clear demand.

4. **Multi-character mood storyboarding.** Storyboard agent emits per-character mood per scene (Maya is angry, Joe is calm in scene 4) instead of one segment-level mood. Improves naturalness of dialogue exchanges.

5. **Per-character lanes in editor (Stage 2).** Real per-track audio with mute / solo / volume. Requires master-buffer split + new export muxer.

6. **AI-suggested mood from script context.** Currently the agent picks mood from segment text; v2 considers context (previous line was angry → reply has elevated tonal continuity). Improves dialogue naturalness.

7. **Subtitle re-timing on stretch (re-Scribe option).** Pay $0.05 to re-run Scribe on stretched audio for exact subtitle alignment vs. lockstep math approximation.

8. **Cross-project mood preferences.** "I always want Maya warm and Joe serious" remembered across projects. Pairs with cross-project cast library (consistency-plan v2).
